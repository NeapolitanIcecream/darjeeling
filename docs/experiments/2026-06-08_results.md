  candidate crate.

## Findings

1. The L4-as-Codex-agent design for L1 is viable in this demo. It produced real
   Rust recognizers, tests passed, and promoted artifacts were benchmarkable.
2. L1 native paths are fast enough to justify the architecture: final promoted
   native p95 was `21.6 us` in the pilot benchmark.
3. L2 is not yet useful at runtime. Training runs, but guard calibration is too
   conservative and accepted no pilot runtime requests.
4. L3 local SLM should remain switchable. It runs on local MPS, but current
   quality and latency are not competitive in this setup.
5. L4 proposal is partially useful. Guard proposals usually succeeded after the
   early fix/retry loop, but L2 proposals often requested unsupported
   `crf/linear_crf`, and L3 prompt proposals consistently failed with empty
   content or connection errors.
6. Workload locality matters strongly. Zipf-heavy streams were much easier for
   this evolutionary cache/program approach than uniform streams.

## Open Issues

- Artifact promotion remains group-level. No run reported
  `promoted_with_layer_regression`, but group promotion can still hide single
  layer regressions and needs a stronger design later.
- L4 proposal needs stricter bounded-output repair. In particular, L2 family
  should be constrained to supported values and parse/semantic failures should
  either retry with feedback or be normalized conservatively.
- L3 prompt optimization did not produce a usable candidate in this run.
- The suite scale is intentionally small. It validates plumbing and directional
  behavior, not statistical stability.
- Gold EM is not the same as teacher self-consistency. Some L4 teacher outputs
  differ from MASSIVE gold, so gold EM should be interpreted as task benchmark
  quality, not as proof that fallback behavior is internally inconsistent.
